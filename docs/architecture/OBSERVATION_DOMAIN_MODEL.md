# ASTRA Observation Domain Model

## Executive Summary

An **Observation** is the central, actionable domain entity within the ASTRA platform. It represents a verified, contextualized security anomaly derived from aggregated telemetry. 
ASTRA explicitly uses *Observations* rather than *Alerts* to prevent alert fatigue. Alerts are often noisy, single-event notifications without context. Observations, conversely, are rich, evidence-backed findings generated only after the system has correlated multiple events and calculated a risk score. 

The Observation fits into the ASTRA pipeline as follows:
`Telemetry → Parser → CES → Correlation → OBSERVATION → Risk Scoring → Policy Evaluation → Action`

---

## Observation Domain Definition

The Observation domain interacts with several surrounding entities:
* **Observation**: The primary unit of investigation representing a potential threat.
* **Correlation**: The grouped incident candidate derived from CES events. An Observation is a promoted Correlation.
* **Event (CES)**: The raw, normalized log line.
* **Policy**: The ruleset evaluated against the Observation to determine the next action.
* **Risk Score**: A quantified metric representing the severity and confidence of the Observation.

**Relationships**:
* One **Correlation** promotes to One **Observation**.
* One **Observation** contains One **Risk Score**.
* One **Observation** is evaluated against Many **Policies** (resulting in one final Policy Action).
* One **Observation** references Many **Events** (via the Correlation).

---

## Observation Lifecycle

The lifecycle dictates how an Observation moves through the system.

**Official Statuses**:
* `NEW`: Initially created by the Observation Engine.
* `POLICY_EVALUATED`: Processed by the Policy Engine with an assigned action.
* `UNDER_REVIEW`: An Incident Responder has begun active investigation.
* `DISMISSED`: The Observation was deemed a false positive or non-actionable.
* `RESOLVED`: The threat was mitigated and the issue closed.

**Valid Transitions**:
* `NEW` → `POLICY_EVALUATED` (System initiated via Policy Engine)
* `POLICY_EVALUATED` → `UNDER_REVIEW` (User initiated via Incident Responder)
* `POLICY_EVALUATED` → `DISMISSED` (User or System initiated)
* `UNDER_REVIEW` → `RESOLVED` (User initiated)
* `UNDER_REVIEW` → `DISMISSED` (User initiated)

**Invalid Transitions**:
* `RESOLVED` → `NEW`
* `DISMISSED` → `UNDER_REVIEW` (Requires new Correlation)
* `NEW` → `RESOLVED` (Must pass through Policy and Review)

## Observation Schema

The official schema for the Observation entity stored in the relational database.

| Field | Data Type | Required | Description |
| :--- | :--- | :--- | :--- |
| `id` | UUID | Yes | Primary key. |
| `title` | String | Yes | Human-readable title generated by the engine. |
| `description` | Text | Yes | Detailed summary of the threat. |
| `correlation_id` | UUID | Yes | Foreign key linking to the originating Correlation. |
| `classification` | String | Yes | Categorization of the threat type. |
| `status` | Enum | Yes | Current lifecycle state. |
| `risk_score` | Integer | Yes | Calculated severity (0-100). |
| `policy_action` | Enum | No | The action recommended by the Policy Engine. |
| `evidence_count` | Integer | Yes | Count of linked CES events. |
| `created_at` | Datetime | Yes | Audit metadata (UTC). |
| `updated_at` | Datetime | Yes | Audit metadata (UTC). |
| `created_by` | String | Yes | System or User UUID. |
| `updated_by` | String | Yes | System or User UUID. |

## Observation Classification

Observations are categorized to assist Responders in filtering and prioritizing work.

* **Authentication**: Login anomalies, brute force, credential stuffing.
* **Privilege Escalation**: Unauthorized role changes, sudo misuse.
* **Network Activity**: Port scanning, C2 beaconing, exfiltration.
* **Malware**: Signature matches, ransomware behavior.
* **Policy Violation**: Unauthorized access to restricted resources.
* **Anomaly**: Statistical deviations from baselines.

*Rationale*: Classifications map directly to MITRE ATT&CK tactics, aiding in downstream case management and reporting.

## Risk Score Standard

The official metric for determining Observation severity.

**Scale**: `0 - 100`

**Score Categories**:
* **INFORMATIONAL** (0-19)
* **LOW** (20-39)
* **MEDIUM** (40-69)
* **HIGH** (70-89)
* **CRITICAL** (90-100)

**Scoring Principles**:
Scores are calculated based on Asset Criticality, Threat Confidence (derived from Correlation rules), and Potential Impact. 

**Scoring Constraints**:
* The score must be deterministic.
* A score cannot exceed 100.
* Manual overrides by users are forbidden; users must change the policy or rule.

---

## Risk Score Examples

* **Brute Force (Failed Logins Only)**: Confidence High, Impact Low = **Score: 35 (LOW)**
* **Suspicious Process (cmd.exe spawning from Word)**: Confidence Medium, Impact High = **Score: 75 (HIGH)**
* **Impossible Travel (Login from US and CN within 5 mins)**: Confidence High, Impact Medium = **Score: 65 (MEDIUM)**
* **Privilege Escalation (Successful Brute Force followed by Admin Role Assignment)**: Confidence High, Impact Critical = **Score: 98 (CRITICAL)**

---

## Policy Action Mapping

Once the Risk Score is generated, the Policy Engine maps the risk to a Recommended Action.

| Risk Category | Recommended Action (PolicyAction Enum) | Rationale |
| :--- | :--- | :--- |
| **Informational** | `OBSERVE` | Keep for historical context; no notification needed. |
| **Low** | `OBSERVE` | Retain in UI for Analysts to review passively. |
| **Medium** | `NOTIFY` | Push to integrations (Slack/Email) for asynchronous review. |
| **High** | `REVIEW_REQUIRED` | Immediately flag for Incident Responder intervention. |
| **Critical** | `REVIEW_REQUIRED` | Flag for immediate intervention (Future: Trigger `FUTURE_MITIGATION`). |

## Observation API Contract

**Base Route**: `/api/v1/observations`

* **GET `/api/v1/observations`**
  * *Purpose*: Fetch a list of observations.
  * *Filters*: `?status=NEW&min_risk=70&classification=Authentication`
  * *Pagination*: `?skip=0&limit=50`
  * *Sorting*: `?sort_by=risk_score&order=desc`
  * *Response*: `{"data": [{Observation Object}, ...], "total": 100}`

* **GET `/api/v1/observations/{id}`**
  * *Purpose*: Fetch full details, including linked event metadata.
  * *Response*: `{"data": {Observation Object with Events array}}`

* **PUT `/api/v1/observations/{id}`**
  * *Purpose*: Update the status (e.g., advancing to `UNDER_REVIEW`).
  * *Body*: `{"status": "UNDER_REVIEW"}`

* **Error Structure**: `{"error": "Validation Failed", "code": 400, "details": "..."}`

## Observation UI Requirements

**Observation List View**:
* *Required Fields*: Title, Classification, Risk Score, Status, Policy Action, Evidence Count, Created At.
* *Required Filters*: Status Dropdown, Risk Category Toggle, Date Range.
* *Role Restrictions*: Viewable by all; write actions limited to Responders.

**Observation Detail View**:
* *Required Fields*: Full description, timeline of events, Risk Score breakdown, Policy evaluation rationale.
* *Required Actions*: "Begin Review", "Dismiss Observation", "Request AI Summary" (Advisory only).

## Performance Requirements

* **Volume**: The system must support processing up to 10,000 new Observations per day.
* **Latency**: The transition from `Correlation` to `POLICY_EVALUATED Observation` must occur within **2 seconds** (Warm Path).
* **Indexing**: DB must be indexed on `status`, `risk_score`, and `created_at` to support UI list rendering in under **500ms** (Cold Path).
* **Retention**: Active observations kept indefinitely; `DISMISSED` and `RESOLVED` observations moved to cold storage after 90 days.

## Future Compatibility

This schema ensures forwards-compatibility with out-of-scope features:
* **Case Management**: `Observation` to `Case` will be a 1-to-1 or Many-to-1 mapping.
* **Automation Engine**: `policy_action` will eventually trigger automated workflows.
* **Recovery Engine**: Will track changes made by Automation linked to the `Observation.id`.
* **Report Engine**: The strict `classification` enum enables clean compliance and executive metric rollups.

---

## Open Risks

1. **False Positives**: Overly broad correlation rules could spam the Observation queue.
2. **Risk Scoring Bias**: Initial algorithms may skew scores too high, causing alert fatigue despite the Observation structure.
3. **Observation Explosion**: A single massive attack could generate thousands of distinct observations instead of one consolidated one.
4. **Storage Growth**: Retaining full event metadata alongside Observations will rapidly consume DB space.
5. **Policy Drift**: Changes to Risk Score algorithms may invalidate historical policy mappings.

---

## Recommendations

* **Immediate (Sprint 3)**: Implement the exact schemas defined here. Treat the Risk Score algorithm as a simple static function for the MVP to prevent delays.
* **Sprint 3**: Ensure the DB uses proper foreign keys for the `correlation_id` to prevent orphaned records.
* **Sprint 4+**: Implement a mechanism to merge overlapping Observations to mitigate the "Observation Explosion" risk.
