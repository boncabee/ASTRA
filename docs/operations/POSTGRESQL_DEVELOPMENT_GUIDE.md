# PostgreSQL Development Guide

## Purpose
This document outlines the architecture, configuration, and workflows associated with the PostgreSQL database in the ASTRA local development environment.

## Scope
This guide covers the Docker-based PostgreSQL setup, database definitions, connection strings, Alembic migration workflows, and common error recovery procedures.

## Procedure

### 1. PostgreSQL Container Architecture
ASTRA utilizes a containerized PostgreSQL instance for local development. This ensures isolation, easy teardown, and consistency across environments. The container exposes port `5432` to the host machine.

### 2. docker-compose Usage
The PostgreSQL service is defined in `docker-compose.yml`.
- **Start the database:** `docker-compose up -d db`
- **Stop the database:** `docker-compose stop db`
- **Tear down the database (and delete volumes):** `docker-compose down -v`

### 3. Database Definitions
The instance provisions two distinct databases:
- **`astra`**: Used for standard local development and manual API testing.
- **`astra_test`**: Used exclusively by `pytest` for integration testing. This database is often truncated or recreated during test runs.

### 4. Connection Strings
Connection strings must use the `asyncpg` driver for asynchronous database access. Ensure these are defined in your `backend/.env` file.
- **Development DB:** `DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5432/astra`
- **Testing DB:** `TEST_DATABASE_URL=postgresql+asyncpg://postgres:postgres@localhost:5432/astra_test`

### 5. Migration Workflow (Alembic)
Alembic is used to manage database schema changes. All commands must be run from the `backend/` directory.
- **Apply all pending migrations:** `alembic upgrade head`
- **Create a new migration:** `alembic revision --autogenerate -m "description_of_changes"`
- **Downgrade one revision:** `alembic downgrade -1`

## Troubleshooting / Recovery Procedures

### InvalidPasswordError
- **Symptoms:** `asyncpg.exceptions.InvalidPasswordError` during application startup or test execution.
- **Root Cause:** Incorrect credentials in `DATABASE_URL` or `TEST_DATABASE_URL`.
- **Resolution:** Verify the username and password match the `POSTGRES_USER` and `POSTGRES_PASSWORD` defined in `docker-compose.yml`. Update `.env` accordingly.
- **Prevention:** Ensure connection strings match documentation explicitly.

### Database does not exist
- **Symptoms:** `asyncpg.exceptions.InvalidCatalogNameError: database "astra_test" does not exist`.
- **Root Cause:** The `astra` or `astra_test` database was not created by the initialization scripts.
- **Resolution:** Restart the container forcing volume recreation: `docker-compose down -v` followed by `docker-compose up -d db`. Alternatively, connect via `psql` and run `CREATE DATABASE astra_test;`.
- **Prevention:** Wait for the PostgreSQL container to fully initialize and run the entrypoint scripts before starting the backend.

### ConnectionRefusedError
- **Symptoms:** Application fails to start with `ConnectionRefusedError` or Alembic hangs.
- **Root Cause:** The Docker container is not running, or port 5432 is occupied by a local PostgreSQL installation.
- **Resolution:** Run `docker ps` to ensure the container is up. If port 5432 is in use, stop your local PostgreSQL service or change the port mapping in `docker-compose.yml`.
- **Prevention:** Use Docker Desktop exclusively for database processes and disable local PostgreSQL Windows services.

### Alembic Migration Failures
- **Symptoms:** `alembic upgrade head` fails with dependency issues or column missing errors.
- **Root Cause:** Target database is out of sync, or autogenerated scripts contain errors.
- **Resolution:** Review the Alembic output. If the database is hopelessly corrupted in local development, the fastest recovery is: `docker-compose down -v`, `docker-compose up -d db`, and `alembic upgrade head`.
- **Prevention:** Always double-check autogenerated Alembic revisions before committing to source control.

## Verification
- Run `alembic upgrade head` successfully without errors.
- Run `pytest` to confirm the `astra_test` database is accessible, properly defined, and functional.
