# Errors Found

- Found a minor risk with timestamp fallback: If an invalid timestamp is provided during partial recovery, it now safely falls back to `datetime.now(timezone.utc).isoformat()` without dropping the event.

# Problems / Observations

1. **Data Loss Risks:** The `raw_event` field in `CESEvent` enforces a `max_length` of 65536 bytes. Extremely large, unsupported log payloads exceeding this limit will trigger a Pydantic validation failure during fallback generation, which may result in data loss if not captured by an external dead-letter queue (DLQ).
2. **Parser Recovery Weaknesses:** The framework is robust, but a complete failure during fallback generation (e.g., severe resource exhaustion or unexpected model constraints) raises a `RecoveryFailureError`. The engine executing the framework must catch this to prevent process crashes.
3. **CES Compatibility Risks:** The `custom.unknown.detected` event type fully complies with the `^([\w-]+)\.([\w-]+)\.([\w-]+)$` regex and the `custom` `EventCategory` enum constraint. No direct incompatibility identified.
4. **Correlation Risks:** Fallback events lack standardized identification artifacts (e.g., source IP, users). This reduces their value to the Correlation Engine. They essentially act as raw storage until a formal parser is built.
5. **Future AKM Risks:** High volumes of `custom.unknown.detected` events could degrade analytics pipeline performance or introduce noise to the Attack Knowledge Model. 

# Recommendations

- Implement a fallback Dead-Letter Queue (DLQ) at the ingestion router level to catch `RecoveryFailureError` and persist logs directly to blob storage as a final safety net.
- Configure alerting to monitor the volume of `custom.unknown.detected` events to quickly identify misconfigurations or the need for new parsers.
- Truncate `raw_event` up to the max size limit automatically in the fallback mapper to prevent validation rejection on extreme log sizes.

# Sprint Impact

The foundation for handling exceptions, unknown formats, and malformed data in the Parser SDK is now secure. Future parsers (VPN, Windows, Firewall) can now safely depend on these fallback abstractions.
