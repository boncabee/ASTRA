# Task Information
- **Objective:** Create a centralized parser registration and discovery mechanism that manages all ASTRA parsers.
- **Goal:** Ensure parser discovery is centralized so that future parsers (VPN, Windows, Firewall, CloudTrail) do not need to be manually instantiated.
- **Coverage Target:** >=95%

# Deliverables Created
- `backend/app/parsers/registry/registry.py`: Core `ParserRegistry` implementation using a thread-safe singleton pattern.
- `backend/app/parsers/registry/exceptions.py`: Custom exceptions (`ParserRegistryError`, `DuplicateParserError`, `ParserNotFoundError`, `InvalidParserError`).
- `backend/app/parsers/registry/__init__.py`: Module exports.
- `backend/tests/parsers/registry/test_registry.py`: Comprehensive test suite for registry lifecycle, validations, and edge cases.

# Validation Results
- **Unit Tests:** 18 passing tests covering all registry capabilities.
- **Coverage:** 100% test coverage achieved for the `app.parsers.registry` module (exceeding the >=95% requirement).
- **Core Capabilities Verified:** Registration, lookup, existence checking, listing, removal, and duplicate/invalid registration blocking.
- **Lifecycle Support:** Metadata configuration (e.g. `name`, `enabled`, `version`) and dynamic enabling/disabling capabilities successfully implemented and validated.
- **Thread Safety:** Implemented `threading.Lock()` to secure all dictionary operations for future concurrent parser ingestion.

# Errors Found
- None during standard execution.

# Problems / Observations
- **Concurrency Risks:** While the registry uses standard Python `threading.Lock()` to manage thread-safety during `dict` updates and lookups, heavy concurrent reads during multi-threaded parser lookups (e.g., hundreds of logs per second checking `get_parser`) could face a minor bottleneck. Currently, a single `_lock` is used for both reads and writes. A future optimization could involve a `ReadWriteLock` if this becomes a measurable performance degradation, or caching the parser class instances on the ingestion workers.
- **Scalability Risks:** If the registry scales to hundreds of plugins (parsers), dynamically loading module dependencies upon registration might cause heavy memory overhead. We should ensure parsers are lazily loaded or only enabled when explicitly configured.
- **Lifecycle Risks:** The registry lacks an implicit hot-reload mechanism. If a parser is added as a plugin during runtime, it requires explicit code registration (via `register_parser`). A future plugin auto-discovery mechanism (e.g., scanning a `plugins/` directory) might be needed.

# Recommendations
- Introduce a dynamic discovery mechanism (e.g., using Python's `entry_points` or directory scanning) in the future to automatically register available parser implementations without explicit code integration.
- For extremely high-throughput ingestion environments, consider providing a fast-path lookup or caching mechanism bypassing the `threading.Lock()` if the registry map remains completely static after initialization.
