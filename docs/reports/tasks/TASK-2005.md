# Golden Dataset Comparison

The parser was validated against `windows_logon_success.json` and `windows_logon_failure.json`. 

**Differences Documented:**
1. **Taxonomy Deviation for User Management:** The original spec requested that Event IDs 4720 and 4726 map to `identity.user.created` and `identity.user.deleted`. However, because `identity` is not an approved member of the `EventCategory` enumeration in `CESEvent`, this triggers a Pydantic `ValidationError`. To respect schema constraints and avoid redesigning the platform, these events were shifted to `custom.user.created` and `custom.user.deleted`.
2. **Metadata Context:** The golden datasets provide contextual metadata (e.g., `process_name`) which the parser correctly maps to `metadata` if it exists in the original log. If missing, it is safely omitted.

# Errors Found

- **Validation Failure:** As mentioned, strict Pydantic enforcement of the `EventCategory` enum caused initial tests for 4720/4726 to fail.

# Problems / Observations

1. **Event ID Ambiguity:** Windows multiplexes vastly different scenarios into single Event IDs. Event 4624 tracks interactive logons, network service logons, and batch operations. Without correlating `LogonType`, mapping all 4624s uniformly to `authentication.login.success` will produce significant noise.
2. **Identity Tracking Risks:** For events like User Creation (4720), Windows provides a `SubjectUserName` (who did it) and a `TargetUserName` (who was created). Mapping `TargetUserName` to `actor.username` complicates security incident investigations, as the true "actor" is the subject.
3. **Domain Controller Variations:** Depending on AD configuration, Domain prefixes can appear in `TargetDomainName`, or appended/prepended to the username (e.g., `CORP\msmith`). This requires a complex enrichment step later to split domain identifiers reliably.
4. **Timezone Issues:** Extracted timestamps frequently lack timezone data. We enforce UTC locally, but local workstations forwarding logs could corrupt the timeline.

# Recommendations

- Create a pre-processing standard for `LogonType` to filter out machine-account noise (LogonType 3/4) from interactive human activity (LogonType 2/10).
- Update the central `CESEvent` schema to formally include `identity` as a first-class `EventCategory`.
- Re-architect the `TransformerConfig` to support conditional field mapping (e.g., mapping `SubjectUserName` instead of `TargetUserName` specifically for administrative events).
