## Storage Design
* The `CorrelationRule` and `CorrelationMatch` SQLAlchemy models accurately mirror the approved Domain Model specifications.
* Added the `expires_at` (DateTime, timezone-aware) column to `CorrelationMatch` to support future data retention pipelines.

## Repository Design
* Created `CRUDCorrelation` in `backend/crud/crud_correlation.py` to decouple business logic from SQLAlchemy execution logic.
* Integrated dynamic, optional filters inside `get_matches()` to efficiently sort matches by time windows and minimum scores.

## Index Strategy
Explicitly indexed columns on `CorrelationMatch`:
1. `rule_id`: Rapid filtering by rule context.
2. `match_timestamp`: Foundational for defining time bounds when searching for related incident events.
3. `correlation_score`: Enables lightning-fast filtration of high severity matches.
4. `created_at` / `expires_at`: Indexed directly via `__table_args__` to accelerate chronological range scans, a mandatory requirement for data cleanup algorithms to execute smoothly over millions of rows.

## Retention Strategy
While scheduled deletion was explicitly excluded from this task's scope, the `expires_at` column is now fully integrated into the architecture. By defaulting to `NULL`, the MVP implicitly retains data indefinitely. Once retention pipelines are built, populating this field guarantees immediate indexing and querying compatibility without structural schema rewrites.
